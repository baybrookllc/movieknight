Supabase + Deno Edge Functions

PostgreSQL 15 with pgvector for AI search. Row-level security enforces per-user data isolation. Eight Deno-runtime edge functions handle external integrations and compute.

• 17 confirmed tables, 31 migrations applied

AI / Search

OpenAI Embeddings + pgvector HNSW

Every title is vectorised with text-embedding-3-small (1 536 dimensions). Semantic search uses an HNSW index for sub-second approximate nearest-neighbour lookups. 8 s timeout with keyword fallback prevents timeouts reaching the client.

• Measured search time: ~740 ms end-to-end
• In-app Claude AI assistant via Anthropic SDK 0.96
• /api/claude/ask supports why_watch, similar, taste, free modes
External APIs

TMDB · DTDD · Anthropic

The app never calls TMDB directly from the client — all requests are routed through the tmdb-cache edge function which caches responses for 7 days (30 days for titles older than 2 years).

• TMDB: movie + TV metadata, posters, episodes
• DTDD: 20+ trigger-warning topics, 30-day cache, 70 % threshold
• Anthropic Claude: in-app AI chat assistant

What Changed & Why

1. Composite indexes on friend_requests

Added (sender_id, status) and (receiver_id, status) indexes. The RPC was doing full sequential scans on a growing table. 60× speedup confirmed.

2. Semantic-search timeout handler

OpenAI embedding calls occasionally exceeded Supabase's default timeout. Added explicit 8-second abort controller with automatic keyword ILIKE fallback — eliminates all 500 errors reaching the client.

3. Sequential RPC batching (connection pool relief)

Three components were firing multiple concurrent RPC calls via Promise.all(), saturating the Supabase connection pool. Calls are now sequential in BadgeProvider, FriendsClient, and ProfilePage.
